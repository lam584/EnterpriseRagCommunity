# 帖子被举报后的审核流程（精简版）

## 1. 目标
- 在帖子已发布后，基于举报信号触发复审并输出稳定处置动作。
- 复审复用帖子主审核链路，仅在前置与策略层做差异化。

## 2. 固定原则
- 举报信号只用于触发、分流、资源调度，不作为违规证据。
- 最终结论必须由内容本身的可验证证据支撑。
- 任意 `REJECT` 缺少可验证证据时，必须改判 `ESCALATE`。
- 仅保留三类提示词：`MODERATION_TEXT`、`MODERATION_VISION`、`MODERATION_JUDGE`。

## 3. 与发布审核的差异（只改这四处）
1) 增加 `report_context` 输入（含 `content_snapshot_id`）。
2) 增加举报前置步骤（去重、限流、聚合、快照、分流）。
3) 使用 `review_stage=reported` 覆盖阈值与升级策略。
4) 将 `decision_suggestion` 映射为复审处置动作。

## 4. 复审流程
### Step 0：举报前置（LLM 前）
### 输入
- `report_id`、`post_id`、`reporter_id`、`reported_at`
- `report_reasons[]`、`report_free_text`（可选）

### 处理
- 去重：同一举报人对同一帖子重复举报只计一次。
- 限流：短时间异常高频举报走风控延迟处理。
- 聚合信号：`report_count_total`、`report_count_unique`、`report_velocity`、`reporter_trust_agg`。
- 冻结快照：生成 `content_snapshot_id`，固化 `title/text/images/hash`。
- 分流队列：
  - `light`：低热度低可信
  - `standard`：达到常规触发阈值
  - `urgent`：高危理由 + 高热度/高可信

### Step 1~3：复用帖子三阶段审核
- 第一阶段：`TextAudit / VisionAudit` 并行取证（超长内容仍可分片）。
- 第二阶段：聚合器规则优先初判。
- 第三阶段：`Judge` 升级仲裁（仅高价值证据子集）。

### 复审额外输入
- `report_context`（必须含 `content_snapshot_id`）
- `review_stage="reported"`（或 `appeal`）

### 复审策略差异
- 更偏召回：高危类目下调 `T_reject`，扩大 `ESCALATE` 区间。
- 更严证据：证据不可验证时禁止 `REJECT`。
- 更强冲突：与历史审核结论冲突时强制升级仲裁或人工。

### Step 4：处置动作映射（策略层）
- `ALLOW`：关闭举报单，恢复临时限制。
- `REJECT`：下架/删除帖子，记录 `evidence_book` 与 `policy_version`。
- `ESCALATE`：进入人工队列，可按配置先折叠/限流/隐藏。

### Step 5：复审闭环（非模型）
- 作者申诉：走同链路，`review_stage="appeal"`。
- 恶意举报治理：降低 `reporter_trust`，必要时进入风控策略。

## 5. 最小可落地配置（建议默认）
- `review_trigger.window_minutes=10`
- 分流阈值：
  - `light`：`report_count_unique >= 3`
  - `standard`：`report_count_unique >= 5` 或 `report_velocity >= 5` 或 `reporter_trust_agg >= 0.7`
  - `urgent`：`report_count_unique >= 20` 或 `report_velocity >= 20` 或（`reporter_trust_agg >= 0.85` 且命中高危理由）
- 复审阈值覆盖：
  - `reported`：`T_allow=0.15`，`T_reject=0.75`
  - `appeal`：`T_allow=0.25`，`T_reject=0.85`
- `require_evidence=true`
- 临时处置时长：最长 60 分钟或直至复审结论返回。
