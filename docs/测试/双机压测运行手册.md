# 双机压测运行手册

## 1. 目标与角色

- 压测机（Load Generator）：运行 JMeter，负责发压。
- 被测机（SUT）：运行 EnterpriseRagCommunity 服务。

推荐拓扑：
- 压测机 A -> 被测机 B:8099

说明：
- 动态端口（ephemeral port）扩容主要做在压测机，因为大量出站连接由压测机发起。
- 被测机通常是监听 8099，不依赖本机临时端口承载入站连接。
- 若被测机还要对外高频发起请求（例如大量回调、外部 API 调用），再考虑同时扩容被测机动态端口。

## 2. 压测前检查

在压测机执行：

- 检查动态端口范围
  - netsh int ipv4 show dynamicport tcp
  - netsh int ipv4 show dynamicport udp
  - netsh int ipv6 show dynamicport tcp
  - netsh int ipv6 show dynamicport udp

- 建议将范围扩大到：
  - StartPort = 1024
  - PortCount = 64511

- 一键脚本（先看、后改）
  - 仅查看：
    - .\scripts\tune-windows-dynamic-ports.ps1
  - 应用修改（管理员 PowerShell）：
    - .\scripts\tune-windows-dynamic-ports.ps1 -Apply -StartPort 1024 -PortCount 64511

可选（谨慎）：
- 降低 TIME_WAIT 持续时间：
  - .\scripts\tune-windows-dynamic-ports.ps1 -Apply -SetTimedWait -TcpTimedWaitDelay 30
- 此项通常需要重启系统生效。

## 3. 被测机启动建议

在被测机使用 perf profile 启动服务：

- java -jar build/libs/EnterpriseRagCommunity.war --spring.profiles.active=perf

当前 perf 配置已做以下降干扰策略：
- 访问日志保留但降采样。
- 请求/响应体抓取关闭。
- 访问日志异步批量写入。

## 4. 压测机执行命令

在压测机仓库根目录执行：

- PowerShell 脚本压测
  - .\perf\jmeter\run-jmeter.ps1 -HostName <被测机IP> -Port 8099 -Threads 3000 -RampSeconds 30 -DurationSeconds 300 -ThinkTimeMs 1000

建议冲刺梯度：
- 3000 -> 3500 -> 4000 -> 4500 -> 5000
- 每档 5 分钟，档间间隔 2~3 分钟。

## 5. 结果采集与判读

每轮结果目录示例：
- perf/jmeter/results/<timestamp>/results.jtl
- perf/jmeter/results/<timestamp>/system-metrics.json

重点指标：
- SuccessRate
- P90 / Avg
- TPS
- TopError 与错误数量
- CPU 与内存可用率

若出现 Address already in use：
- 优先检查压测机端口资源。
- 查看 TIME_WAIT 是否异常升高。
- 降低单机并发并改为多压测机分摊。

## 6. 访问日志在压测中的配置口径

非压测默认口径（application.properties）：
- sample-rate=1.0
- async-enabled=false

压测口径（application-perf.properties）：
- sample-rate=0.10
- keep-error=true
- async-enabled=true
- async-batch-size=500
- async-flush-interval-ms=100
- async-queue-capacity=120000
- async-drop-when-full=true

解释：
- 压测时保留访问日志能力，不是关闭功能。
- 常规请求采样写入，5xx 仍尽量保留。
- 通过异步批量写入降低对业务请求线程的干扰。

## 7. 故障排查清单

- 压测机
  - 动态端口是否已扩容
  - TIME_WAIT 是否过高
  - 本机是否同时运行重负载任务

- 被测机
  - Tomcat 线程池和连接数是否到达上限
  - 数据库连接池是否打满
  - 关键接口是否出现慢 SQL

- 网络
  - 两机之间是否有丢包或带宽瓶颈

## 8. 推荐实践

- 论文与报告采用双口径：
  - 业务极限口径（perf profile）
  - 全日志口径（默认 profile）
- 同时保留可复现命令和配置快照，便于答辩与复核。
