# 帖子评论被举报后的审核流程（精简版）

## 1. 目标
- 评论发布后被举报时，快速进入复审并产出可执行处置结果。
- 复审复用评论主审核链路，只保留“新增差异”。

## 2. 固定原则
- 举报信号只用于触发与分流，不作为违规证据。
- 复审结论必须引用可验证证据（评论原文、OCR、图片编号）。
- 任意 `REJECT` 缺证据时必须 `ESCALATE`。
- 仅保留三类提示词：`MODERATION_TEXT`、`MODERATION_VISION`、`MODERATION_JUDGE`。

## 3. 与发布审核的差异（只改这四处）
1) 输入新增 `report_context`（含 `content_snapshot_id`）。
2) 新增举报前置（去重、限流、聚合、快照、分流）。
3) `review_stage=reported` 下执行复审阈值覆盖与更严升级规则。
4) 策略层把 `decision_suggestion` 映射为复审处置动作。

## 4. 复审流程
### Step 0：举报前置（LLM 前）
### 输入
- `report_id`、`comment_id`、`post_id`、`reporter_id`、`reported_at`
- `report_reasons[]`、`report_free_text`（可选）

### 处理
- 去重：同一举报人对同一评论重复举报只计一次。
- 限流：短时异常高频举报走风控延迟处理。
- 聚合信号：`report_count_total`、`report_count_unique`、`report_velocity`、`reporter_trust_agg`。
- 冻结快照：生成 `content_snapshot_id`，固化：
  - `comment_text`
  - `comment_images`（含 hash）
  - 必要 `thread_context`（如 `replied_comment_excerpt`、`post_excerpt`）
- 分流：`light` / `standard` / `urgent`。

### Step 1~3：复用评论三阶段审核
- 第一阶段：`TextAudit / VisionAudit` 并行取证。
- 第二阶段：聚合器规则优先初判。
- 第三阶段：`Judge` 升级仲裁（仅高价值证据子集）。

### 复审额外输入
- `report_context`（必须含 `content_snapshot_id`）
- `review_stage="reported"`（或 `appeal`）

### 复审策略差异
- 更偏召回：高危类目下调 `T_reject`，扩大 `ESCALATE` 区间。
- 更严证据：举报理由不能替代内容证据。
- 更强冲突：与历史结论冲突时强制升级仲裁或人工。

### Step 4：处置动作映射（策略层）
- `ALLOW`：关闭举报单，恢复临时限制。
- `REJECT`：隐藏/删除评论；记录 `evidence_book` 与 `policy_version`。
- `ESCALATE`：进入人工队列；携带证据簿、图片/OCR 证据、线程上下文摘要。

### Step 5：复审闭环（非模型）
- 申诉：作者申诉走同链路，`review_stage="appeal"`。
- 恶意举报治理：下调 `reporter_trust`，必要时接入风控策略。

## 5. 最小可落地配置（建议默认）
- `review_trigger.window_minutes=10`
- 分流阈值：
  - `light`：`report_count_unique >= 3`
  - `standard`：`report_count_unique >= 5` 或 `report_velocity >= 5` 或 `reporter_trust_agg >= 0.7`
  - `urgent`：`report_count_unique >= 20` 或 `report_velocity >= 20` 或（`reporter_trust_agg >= 0.85` 且命中高危理由）
- 复审阈值覆盖：
  - `reported`：`T_allow=0.15`，`T_reject=0.75`
  - `appeal`：`T_allow=0.25`，`T_reject=0.85`
- `require_evidence=true`
- 临时处置时长：最长 60 分钟或直至复审结论返回。
