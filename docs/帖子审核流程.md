# 帖子审核流程（精简版）

## 1. 目标
- 统一帖子审核主链路，覆盖纯文本、图文、超长图文与被举报复审。
- 保持统一输出：`decision_suggestion = ALLOW | REJECT | ESCALATE` + 可验证 `evidence`。
- 策略层负责最终处置动作，模型层只负责风险建议与证据。

## 2. 固定口径
### 2.1 提示词
- 仅保留 3 个 `prompt_code`：
  - `MODERATION_TEXT`
  - `MODERATION_VISION`
  - `MODERATION_JUDGE`
- 升级仲裁、灰区复核、分片终裁统一复用 `MODERATION_JUDGE`。
- 历史冗余提示词（`MODERATION_JUDGE_UPGRADE`、`MODERATION_TEXT_UPGRADE`、`MODERATION_FINAL_REVIEW`）已下线，不再作为配置入口。

### 2.2 核心原则
- 举报信号仅用于触发与分流，不作为违规证据。
- 任意 `REJECT` 必须有至少 1 条可验证证据。
- 证据不可验证、证据冲突、上下文不足时必须 `ESCALATE`。

## 3. 输入与输出契约
### 3.1 输入（LLM 前先整理）
- `post_id`
- `title`（可空）
- `text`（可空）
- `images[]`（可空，含 url/bytes/hash/元信息）
- `lang`（可选）
- `user_context`（可选）
- `report_context`（仅复审场景）：
  - `report_id`、`reported_at`、`report_reasons[]`
  - `report_count_total`、`report_count_unique`、`report_velocity`
  - `reporter_trust_agg`
  - `content_snapshot_id`（复审必填）
- `policy_version`
- `review_stage`（可选：`reported` / `appeal`）

### 3.2 输出（统一结构）
- `risk_score`（0~1）
- `decision_suggestion`（`ALLOW` / `REJECT` / `ESCALATE`）
- `labels[]`
- `severity`（`LOW` / `MID` / `HIGH` / `CRITICAL`）
- `evidence[]`（必须可定位到文本片段、OCR 片段或图片编号）
- `uncertainty`（参考信号）
- `notes`（系统内部说明）

## 4. 路由规则
- 文本为主：`TextAudit` 优先。
- 图片为主：`VisionAudit` + OCR，必要时与 `TextAudit` 并行。
- 图文融合与终裁：`Judge` 负责最终融合判定。
- 被举报复审：复用同一三阶段流水线，仅增加举报前置与复审阈值覆盖。

## 5. 三阶段流水线
### Step 0：预处理（LLM 前）
- 规范化文本并保留原文映射，避免证据无法回溯。
- 提取轻量规则特征（联系方式、引流模板、规避写法、重复度）。
- 图像做去重与 OCR 预处理；超量图片按风险优先进入深度复核。

### Step 1：并行取证
- `TextAudit`：输出文本风险、标签、证据、不确定性。
- `VisionAudit`：逐图输出图像风险、标签、OCR、证据、视觉信号。
- 证据必须可定位；不可定位视为无效证据。

### Step 2：聚合初判（规则优先）
- 合并多源结果得到 `post_risk_max` 与候选标签。
- 先做硬门槛校验：证据、冲突、OCR 质量、上下文完整性。
- 给出初判：`ALLOW` / `REJECT` / `ESCALATE`。

### Step 3：升级仲裁（仅少量样本）
- 触发条件：
  - 结论冲突
  - 证据不足但风险偏高
  - 跨片语义依赖明显
  - 复审场景与历史结论冲突
- 仅输入高价值子集：Top-K 高风险分片、Top-K 高可疑分片、Top-K 高风险图片与证据簿。
- 仲裁后仍无法形成可验证证据时，必须 `ESCALATE`。

## 6. 决策规则（策略层）
### 6.1 阈值合成顺序
1) 先取场景默认阈值：`thresholds.default`，若命中 `by_review_stage` 则覆盖。
2) 再按 `labels[]` 应用 `by_label` 覆盖。
3) 多标签同时命中时取更严格阈值（`min`）。

### 6.2 硬门槛
- `REJECT` 前提：
  - 至少 1 条可验证 `evidence`
  - 规则校验通过
- 以下情况一律 `ESCALATE`：
  - 证据为空或不可验证
  - 结论依赖 OCR 但 OCR 证据质量不足
  - 多来源结论冲突且无法消解
  - 跨片语义不足以支撑最终结论

## 7. 四类场景最短路径
- 短纯文本：`TextAudit -> 策略层 -> 必要时 Judge`
- 短图文：`VisionAudit + TextAudit -> Judge -> 策略层`
- 超长图文：`语义分片 TextAudit + VisionAudit -> 聚合 -> Judge`
- 被举报复审：`举报前置 -> 复用以上路径 -> 复审动作映射`

## 8. 被举报复审（review_stage = reported）
### Step 0：举报前置
- 事件去重：同一举报人重复举报合并。
- 聚合信号：`report_count_unique`、`report_velocity`、`reporter_trust_agg`。
- 冻结快照：生成 `content_snapshot_id` 并固化内容。
- 分流队列：`light` / `standard` / `urgent`。

### 复审策略差异
- 提升召回：下调高危类目 `T_reject`，扩大 `ESCALATE` 区间。
- 加强证据：禁止用举报理由替代内容证据。
- 强化冲突处理：与历史结论冲突时强制升级仲裁或人工。

## 9. 处置动作映射
- `ALLOW`：内容保持可见；若有临时限制则恢复。
- `REJECT`：下架/删除帖子；记录 `evidence_book` 与 `policy_version`。
- `ESCALATE`：进入人工队列；可按配置先折叠/限流/隐藏。

## 10. 最小可落地配置
- `thresholds.default`：`T_allow=0.20`，`T_reject=0.80`
- `thresholds.by_review_stage.reported`：`T_allow=0.15`，`T_reject=0.75`
- `thresholds.by_review_stage.appeal`：`T_allow=0.25`，`T_reject=0.85`
- `escalate_rules.require_evidence=true`
- `hard_reject_labels`：高危类目按业务配置
