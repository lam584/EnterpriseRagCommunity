# 帖子评论审核流程（精简版）

## 1. 目标
- 覆盖两类触发：`comment_stage=publish` 与 `review_stage=reported`。
- 保持统一输出：`ALLOW | REJECT | ESCALATE` + 可验证证据。
- 与帖子主流程共用同一骨架，仅保留评论差异。

## 2. 固定口径
### 2.1 提示词
- 仅保留：
  - `MODERATION_TEXT`
  - `MODERATION_VISION`
  - `MODERATION_JUDGE`

### 2.2 评论场景硬规则
- 举报信号只用于触发复审，不作为违规证据。
- 任意 `REJECT` 必须有可验证证据（评论原文、OCR、图片编号）。
- 证据不足、证据冲突、上下文缺失时必须 `ESCALATE`。

## 3. 输入与输出
### 3.1 输入
- `comment_id`、`post_id`、`author_id`
- `parent_comment_id`、`root_comment_id`（可空）
- `created_at`
- `text`（可空）
- `images[]`（可空）
- `thread_context`（可选，最小必要）：
  - `post_title`
  - `post_excerpt`
  - `replied_comment_excerpt`
  - `conversation_window`
- `user_context`（可选）
- `report_context`（复审场景可选，含 `content_snapshot_id`）
- `policy_version`
- `comment_stage` 或 `review_stage`（二选一）

### 3.2 输出
- `risk_score`、`decision_suggestion`、`labels[]`、`severity`
- `evidence[]`（可定位、可验证）
- `uncertainty`、`notes`

## 4. 路由规则
- 纯文本评论：`TextAudit` 为主。
- 图文评论：`VisionAudit + TextAudit` 并行，`Judge` 融合。
- 复审场景：不新增模型链路，仅在前置与策略层做差异化。

## 5. 三阶段流水线
### Step 0：预处理（LLM 前）
- 规范化文本，保留原文映射，防止证据回溯失败。
- 提取轻量反垃圾特征（重复、模板引流、规避写法）。
- 按最小原则构建 `thread_context`，避免把整楼全量喂给模型。

### Step 1：并行取证
- `TextAudit`：输入 `text + thread_context + user_context`。
- `VisionAudit`：逐图输出风险、标签、OCR 与证据。
- 要求：证据必须可定位到输入原文/OCR/图片。

### Step 2：聚合初判
- `comment_risk_max = max(text_risk, image_risk_max)`。
- 先做硬门槛校验，再给出 `ALLOW/REJECT/ESCALATE`。
- 典型强制升级：
  - 高分但证据不可验证
  - 依赖 OCR 但 OCR 证据质量不足
  - 依赖上下文但 `thread_context` 缺失
  - 文本与视觉结论冲突

### Step 3：升级仲裁
- 触发条件：
  - 冲突未消解
  - 证据不足但风险偏高
  - 复审场景与历史结论冲突
- 升级后仍无法形成可验证证据时，必须 `ESCALATE`。

## 6. 决策规则
### 6.1 阈值生效顺序
1) 先取 `thresholds.default` 或 `thresholds.by_review_stage[review_stage].default`。
2) 再按 `labels[]` 应用 `thresholds.by_label`。
3) 多标签取更严格阈值（`min`）。

### 6.2 硬门槛
- 仅当 `risk_score >= T_reject` 且证据可验证时才允许 `REJECT`。
- 以下情况必须 `ESCALATE`：
  - 证据为空或不可验证
  - 依赖上下文但缺失 `replied_comment_excerpt`
  - OCR 证据质量不足
  - 规则校验失败或多源冲突

## 7. 发布审核（comment_stage=publish）
- 默认路径：`TextAudit -> 策略层 -> 必要时 Judge`。
- 图文路径：`VisionAudit + TextAudit -> Judge -> 策略层`。
- 动作映射：
  - `ALLOW`：评论可见
  - `REJECT`：隐藏/删除评论，可选折叠回复链或限制互动
  - `ESCALATE`：进入人工队列，可临时隐藏/折叠

## 8. 被举报复审（review_stage=reported）
### Step 0：举报前置
- 去重与限流：合并重复举报，抑制短时异常。
- 聚合信号：`report_count_unique`、`report_velocity`、`reporter_trust_agg`。
- 冻结快照：生成 `content_snapshot_id`，固化评论文本、图片与必要线程上下文。
- 分流：`light` / `standard` / `urgent`。

### 复审策略差异
- 更偏召回：下调高危类目 `T_reject`，扩大 `ESCALATE` 区间。
- 更严证据：禁止用举报理由代替内容证据。
- 更强冲突处理：与历史结论冲突时强制升级仲裁或人工。

### 复审动作映射
- `ALLOW`：关闭举报单，恢复临时限制。
- `REJECT`：隐藏/删除评论并记录 `evidence_book` 与 `policy_version`。
- `ESCALATE`：进入人工队列，附完整证据簿与上下文摘要。

## 9. 最小可落地配置
- `thresholds.default`：`T_allow=0.20`，`T_reject=0.80`
- `by_review_stage.reported`：`T_allow=0.15`，`T_reject=0.75`
- `by_review_stage.appeal`：`T_allow=0.25`，`T_reject=0.85`
- `escalate_rules.require_evidence=true`
- `thread_context_limits`：按最小必要上下文配置
- `anti_spam_rules`：评论频控与相似度规则前置执行
