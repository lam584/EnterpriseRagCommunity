# 自动化测试与指标说明

## 一键运行

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\test-all.ps1
```

可选参数：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\test-all.ps1 -SkipSecurity
powershell -ExecutionPolicy Bypass -File .\scripts\test-all.ps1 -SkipPerf
```

产物默认输出到 `test-reports/`，并生成 `test-reports/SUMMARY.md` 作为总览入口（脚本会把各工具的原始报告复制/汇总到该目录，便于打包归档）。

## 功能测试（回归/冒烟）

功能测试主要由后端 `test`/`integrationTest` 与前端 Vitest 用例构成，用于验证接口/组件在给定输入与权限下的正确性与回归风险（各类运行方式与报告产物见上表）。

- 当前已知排除项：`my-vite-app/src/services/postService.test.ts` 被 Vitest 配置排除，不参与用例统计与覆盖率计算
- 推荐指标：用例数、通过率、失败用例列表（按模块/接口分类）、覆盖的关键业务路径（登录、权限、发帖/评论、检索/问答等）

## 覆盖面怎么描述（建议写到论文/报告里的口径）

覆盖面建议同时从“业务维度”和“技术维度”给出清单，让读者能明确知道：哪些链路被自动化保障了，哪些仍需要手工/补测。

### 业务维度（覆盖到哪些关键链路）

- 账号与安全：注册/管理员初始化、权限守卫、2FA（TOTP）相关策略与页面逻辑、匿名/登录态访问控制
- 内容社区：帖子详情与评论展示、收藏/书签等内容相关接口的权限与行为
- 检索/问答：门户检索页面交互与跳转、后端检索/引用等相关接口与服务逻辑的正确性
- 审计与运维：审计日志接口、访问日志/审计 Filter 行为、管理端队列/监控相关接口安全性
- AI/RAG 相关：路由配置、rerank 映射、提示词/策略解析、token 计数与成本计算等“规则/解析类”逻辑

### 技术维度（覆盖到哪些层）

- 后端：Web 层（Controller）、安全层（Spring Security/Filter）、业务层（Service）、数据与迁移（DTO 校验/SQL 生成/迁移规则）
- 前端：页面与组件（渲染/交互/路由）、服务层（service 方法）、纯工具函数（utils）

### 不在覆盖范围/需要注意的空白（如实说明更加分）

- 端到端（E2E）覆盖：当前默认不包含真实浏览器驱动的 E2E（例如完整“打开页面→登录→发帖→评论→检索”的全链路），更多是组件/接口级自动化
- 外部依赖：涉及第三方服务（例如外部 LLM/向量库/外部鉴权等）的场景通常会在单测里被 mock 或弱化验证；需要在集成/环境联调中补足
- 压测与安全扫描：属于非功能/安全保障手段，不能替代功能正确性的回归测试（反之亦然）

## 高并发/性能测试

powershell -ExecutionPolicy Bypass -File .\perf\jmeter\run-jmeter.ps1 -HostName localhost -Port 8099 -Protocol http -Threads 500 -RampSeconds 5 -Loops 600

- 计划文件：`perf/jmeter/EnterpriseRagCommunity_basic_load.jmx`（运行方式与报告产物见上表）
- 常用命令（示例）：`powershell -ExecutionPolicy Bypass -File .\perf\jmeter\run-jmeter.ps1 -HostName localhost -Port 8099 -Protocol http -Threads 50 -RampSeconds 10 -Loops 20`
- 如果 `results.jtl` 只有表头（例如你看到 `timeStamp,elapsed,label,...` 这一行）：
  - 先看同目录下的 `jmeter.log`（脚本会输出到 `perf/jmeter/results/<时间戳>/jmeter.log`），通常能直接看到 Java/JMeter 启动失败、目标不可达、TLS/协议不匹配等原因
  - 确认被压测服务已启动且可访问：`http(s)://<host>:<port>/api/public/site-config`
  - 确认 `Threads/Loops` 都大于 0；若目标是 HTTPS，务必传 `-Protocol https`（或在 JMX 中设置 `${__P(protocol,...)}` 对应参数）

推荐指标（毕业设计常用口径）：
- 吞吐量（Throughput / TPS）
- 平均响应时间、P90/P95/P99
- 错误率（Error %）
- 并发数（Threads / Users）

## 服务端资源指标（压测配套）

- 通过 `application-perf.properties` 启用 Prometheus 指标端点（默认仅本机 127.0.0.1）
  - 指标端口：`http://127.0.0.1:8098/actuator/prometheus`
- 推荐在压测期间用 Prometheus+Grafana 或其他采集方式记录：
  - CPU、内存、GC、线程数
  - HTTP 请求耗时分布（如果有相应 Micrometer 指标）

## 安全测试

- 报告产物路径见上表

### SCA（依赖漏洞）
再次运行：
```powershell
.\gradlew.bat dependencyCheckAnalyze --no-daemon --rerun
```
需要更新/清理 NVD 数据库（可选）
```powershell
# 更新 NVD 数据
.\gradlew.bat dependencyCheckUpdate --no-daemon

# 清理本地 NVD 缓存（再跑一次分析会重新下载/构建数据）
.\gradlew.bat dependencyCheckPurge --no-daemon
.\gradlew.bat dependencyCheckAnalyze --no-daemon --rerun
```

推荐指标：
- 漏洞数（按严重等级 Critical/High/Medium/Low）
- 漏洞类型（可映射 OWASP Top 10）
- 依赖漏洞清单（CVE/组件版本/修复建议）
