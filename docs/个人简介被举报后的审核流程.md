# 个人简介被举报后的审核流程（精简版）

## 1. 目标
- 在个人资料已生效后，基于举报触发复审并输出最小化处置结果。
- 复审复用个人简介主审核链路，仅保留“新增差异”。

## 2. 固定原则
- 举报信号只用于触发、分流、调度，不作为违规证据。
- 复审结论必须由可验证证据支撑（字段原文、OCR、图片编号）。
- `REJECT` 缺证据时必须改为 `ESCALATE`。
- 提示词仅保留：`MODERATION_TEXT`、`MODERATION_VISION`、`MODERATION_JUDGE`。

## 3. 与资料更新审核的差异（只改这四处）
1) 输入新增 `report_context`（含 `content_snapshot_id`）。
2) 新增举报前置（去重、限流、聚合、快照、分流）。
3) `review_stage=reported` 覆盖阈值并强化升级。
4) `decision_suggestion` 映射为字段级最小处置动作。

## 4. 复审流程
### Step 0：举报前置（LLM 前）
### 输入
- `report_id`、`user_id`、`profile_id`（可选）、`reporter_id`、`reported_at`
- `report_reasons[]`、`report_free_text`（可选）

### 处理
- 去重：同一举报人对同一资料重复举报只计一次。
- 限流：短时异常高频举报进入风控延迟处理。
- 聚合信号：`report_count_total`、`report_count_unique`、`report_velocity`、`reporter_trust_agg`。
- 冻结快照：生成 `content_snapshot_id`，固化：
  - `profile_fields`（`username/bio/location/website/avatar_url`）
  - `avatar_hash` 与头像元信息
  - `old_profile_fields`（用于回滚与冲突判断）
- 分流：`light` / `standard` / `urgent`。

### Step 1~3：复用个人简介三阶段审核
- 第一阶段：`TextAudit / VisionAudit` 并行取证。
- 第二阶段：聚合器规则优先初判。
- 第三阶段：`Judge` 升级仲裁（只输入高价值证据）。

### 复审额外输入
- `report_context`（必须含 `content_snapshot_id`）
- `review_stage="reported"`（或 `appeal`）

### 复审策略差异
- 更偏召回：高危类目下调 `T_reject`，扩大 `ESCALATE` 区间。
- 更严证据：证据不可验证时禁止 `REJECT`。
- 更强冲突：与历史审核/当前生效版本冲突时强制升级仲裁或人工。

### Step 4：处置动作映射（策略层）
- `ALLOW`：关闭举报单，恢复临时隐藏或回滚状态。
- `REJECT`：优先字段级最小处置：
  - 清空 `bio/website`
  - 回滚昵称
  - 撤下头像并恢复默认头像
  - 必要时叠加账号处罚（按业务配置）
- `ESCALATE`：进入人工队列；携带证据簿、快照信息与字段级建议动作。

### Step 5：复审闭环（非模型）
- 申诉：走同链路，`review_stage="appeal"`。
- 恶意举报治理：下调 `reporter_trust`，必要时进入风控策略。

## 5. 最小可落地配置（建议默认）
- `review_trigger.window_minutes=10`
- 分流阈值：
  - `light`：`report_count_unique >= 3`
  - `standard`：`report_count_unique >= 5` 或 `report_velocity >= 5` 或 `reporter_trust_agg >= 0.7`
  - `urgent`：`report_count_unique >= 20` 或 `report_velocity >= 20` 或（`reporter_trust_agg >= 0.85` 且命中高危理由）
- 复审阈值覆盖：
  - `reported`：`T_allow=0.15`，`T_reject=0.75`
  - `appeal`：`T_allow=0.25`，`T_reject=0.85`
- `require_evidence=true`
- 临时处置时长：最长 60 分钟或直至复审结论返回。
